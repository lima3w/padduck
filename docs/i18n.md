# Internationalization (i18n)

Padduck's UI strings are sourced from locale files rather than hardcoded in components, so adding a new language requires no code changes — just a new JSON file.

## How it works

- The frontend uses [i18next](https://www.i18next.com/) + [react-i18next](https://react.i18next.com/).
- Locale files live in `frontend/src/locales/<lang>.json` (e.g. `en.json`, `fr.json`) and are bundled directly at build time — no runtime fetch.
- `en.json` is the reference locale: every translatable string in the app has a key there.
- Any locale file may be **incomplete**. Missing keys automatically fall back to the English (`en`) value, so a partial translation is always safe to ship.
- The user's preferred locale is persisted to their account (`PUT /api/v1/auth/me/locale`) and cached in `localStorage` as a fallback for logged-out/offline use. It can be changed from **Account Settings → Profile → Language**.

## Locale file structure

Keys are nested by feature area:

```json
{
  "nav": {
    "dashboard": "Dashboard",
    "networks": "Networks"
  },
  "login": {
    "signIn": "Sign In"
  }
}
```

Components read strings with the `t()` function from `react-i18next`'s `useTranslation()` hook, using dot-separated paths:

```jsx
import { useTranslation } from 'react-i18next'

function Example() {
  const { t } = useTranslation()
  return <button>{t('login.signIn')}</button>
}
```

## Adding a new language

1. Copy `frontend/src/locales/en.json` to `frontend/src/locales/<code>.json`, where `<code>` is a language code (e.g. `de`, `es`, `pt-BR`).
2. Translate the values — leave keys unchanged. You don't need to translate every key; anything left out (or left untranslated) falls back to English.
3. Register the new locale in `frontend/src/i18n.js` (add it to the `resources` map) and in `AVAILABLE_LOCALES` in the same file, so it shows up in the language switcher.
4. Run the frontend test suite (`npm test`) and open the app locally to sanity-check layout — some languages produce longer strings than English and may need CSS adjustments in extreme cases.

## Adding a new translatable string

When adding new UI text:

1. Add the key and English value to `frontend/src/locales/en.json`.
2. Use `t('your.new.key')` in the component instead of a hardcoded string.
3. Optionally add translations to other locale files — if you don't, the string just falls back to English for those languages until someone translates it.

## Current coverage

As of this writing, the shared app chrome (navigation, header, user menu), the login flow, and the account settings page are fully translated. Other pages are being migrated incrementally — if you're touching a page that still has hardcoded strings, converting it to use `t()` as part of your change is welcome but not required.
